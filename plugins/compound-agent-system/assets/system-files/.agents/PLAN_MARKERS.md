# PLAN_MARKERS — Activation Markers in Planning Documents

> Plan documents can declare Compound-Protocol metadata that the harness reads to
> pre-populate `TASKS.json`. Two forms: **frontmatter** (YAML) for structured plans,
> **inline** (single-line tags) for prose plans.

---

## Why

A plan written without harness-awareness has all the information (goal per phase, DoD
implicit in success criteria, skills implicit in nature of work) but the agent has to
re-extract it on every read. Activation markers make the plan **machine-readable**:

- `node .agents/task.mjs import task_plan.md` → reads markers → creates tasks in ledger
- Each phase becomes a task with id, goal, DoD, declared skills
- Plan stays the source of truth; ledger reflects current execution state

---

## Form 1: YAML frontmatter

For structured plans (numbered phases, explicit DoD):

```yaml
---
compound: active
project: ecc-browser
phases:
  - id: phase-3
    goal: "Build E2E test suite for basket flow"
    dod:
      - check: test
        command: "npx playwright test tests/e2e/"
      - check: artifact
        path: "tests/e2e/basket.spec.js"
      - check: manual
        description: "5 specs cover all basket interactions"
    skills:
      - "e2e-testing"
      - "playwright"
    parent: null

  - id: phase-3.1
    goal: "Wire CI to run E2E on PR"
    dod:
      - check: artifact
        path: ".github/workflows/test.yml"
        contains: "playwright"
    skills:
      - "github-ops"
    parent: "phase-3"
---

# Plan: ECC-Browser Phase 3

## Phase 3 — E2E Test Suite

(human-readable plan content here, identical to today's plans)
```

Import:
```bash
node .agents/task.mjs import task_plan.md
# Reads frontmatter, creates 2 tasks (phase-3 + phase-3.1).
# Existing tasks with same id are skipped (no duplicates).
```

## Form 2: Inline markers

For prose-style plans without structured frontmatter:

```markdown
# Roadmap

## Sprint 1: Foundation

[COMPOUND-PHASE id=foundation goal="Set up build pipeline" dod="test:npm test;artifact:dist/index.html" skills="bun-runtime"]

We're going to set up the basic build pipeline using esbuild...

## Sprint 2: Features

[COMPOUND-PHASE id=features goal="Add three core features" dod="test:node --test tests/" skills="frontend-patterns" parent=foundation]

The three features are...
```

Import same as above. Inline markers and frontmatter can coexist — the harness reads both.

---

## Marker syntax (inline form)

```
[COMPOUND-PHASE id=<id> goal="<text>" dod="<spec>;<spec>" skills="<a>;<b>" parent=<id>]
```

- `id` — unique within the plan; matches `TASKS.json` task id
- `goal` — quoted string; multiline not supported (use frontmatter for that)
- `dod` — semicolon-separated specs, each in form `<type>:<value>`:
  - `test:<command>` — e.g., `test:npm test`
  - `artifact:<path>` — e.g., `artifact:dist/foo.html`
  - `manual:<description>` — e.g., `manual:Browser opens correctly`
- `skills` — semicolon-separated skill ids
- `parent` — optional parent task id (for sub-tasks)

## Diff mode

When a plan is updated and re-imported, the harness can show diffs:

```bash
node .agents/task.mjs import task_plan.md --diff
# Reports:
#   + phase-4 (new in plan)
#   ~ phase-3 (DoD changed: added check: manual "5 specs ...")
#   - phase-2 (in ledger but not in plan; suggest park or remove from plan)
```

Operator reviews and applies with `--apply` or hand-reconciles. Default is
**dry-run**: no changes without `--apply` flag.

---

## Activation marker (whole-document)

The line `compound: active` (in YAML frontmatter) or a standalone marker
`<!-- COMPOUND: active -->` near the top of a plan signals: this plan is
harness-aware and `import` is supported. Plans without this marker are
ignored by `task import`.

This is intentional — not every markdown file in a repo is a plan. The marker
is opt-in per file.

---

## When to use which form

| Scenario | Recommended form |
|---|---|
| New plan, you're writing it now | YAML frontmatter |
| Existing plan, retrofitting | Inline markers (less invasive) |
| Plan auto-generated by `assemble.mjs` KICKOFF | YAML frontmatter (assembler emits it) |
| Plan in a wiki/doc tool that strips frontmatter | Inline markers |

---

## Failure modes

- **Marker in wrong place** → Parser ignores it. `task import --strict` errors on unparseable markers.
- **Plan and ledger drift** → `task import --diff` surfaces drift. Operator reconciles.
- **Phase split into sub-tasks during execution** → Add to plan with `parent` referencing original id. Re-import.
- **Skill referenced in marker doesn't exist** → `task import` warns; per `SKILL_SELECT.md`, you must create the skill or pick a different one before opening the task.
